# Демо-стенд для показа заказчику: Vercel + Neon + Vercel Blob

Публичная ссылка на копию сайта с выборкой ~150 материалов. Аккаунта нужно два —
**Vercel** и **Neon**, карта не нужна нигде.

Зачем не два сервиса, а три: **Neon** — только Postgres, файлы он не хранит. Наши фото
лежат в MinIO, который на Vercel не поднять (там нет ни контейнеров, ни диска). Поэтому
картинки уезжают в **Vercel Blob** — это хранилище внутри того же аккаунта Vercel,
на тарифе Hobby бесплатное (включено 5 ГБ, нужно ~90 МБ).

## Что уже сделано в коде

- `src/lib/storage.ts` — хранилище за одним интерфейсом с двумя движками: MinIO локально,
  Blob на Vercel. Движок выбирается сам по наличию `BLOB_READ_WRITE_TOKEN`, который
  платформа подставляет автоматически. Локально его нет → работает MinIO, docker compose
  и все скрипты миграции не меняются.
- `next.config.ts` — `output: "standalone"` теперь только для Docker (`BUILD_TARGET=docker`
  выставляет Dockerfile). На Vercel он ломает сборку: платформа сама пакует функции.
- `NEXT_PUBLIC_IS_DEMO=1` закрывает демо от индексации (метатег robots + robots.txt).
- `scripts/export-demo.ts` (`npm run export-demo`) — перелив выборки в Neon + Blob.

## Шаг 1. Neon (база)

1. Регистрация на neon.com, новый проект, регион — Frankfurt (ближе к РФ).
2. Скопировать строку подключения (Connection string, с `?sslmode=require`).
3. Локально в `.env` добавить: `DEMO_DATABASE_URL=<строка>`.
4. Накатить схему на Neon (PowerShell, из корня проекта):
   ```powershell
   $env:DATABASE_URL="<строка Neon>"; npx prisma migrate deploy
   ```
   Переменная живёт только в этом окне терминала и локальный `.env` не трогает — в
   следующем окне `DATABASE_URL` снова будет локальным.
   Проверка: в Neon SQL Editor `select count(*) from "Section";` → таблица есть, 0 строк.

Бесплатный тариф — 0.5 ГБ. У нас только тексты, влезает с запасом.

## Шаг 2. Vercel: проект и хранилище Blob

1. Регистрация на vercel.com через GitHub → Add New Project → импортировать `r0k11/esque`.
2. Framework Next.js определится сам. Build command и output — не трогать.
3. Первый деплой упадёт или соберётся без данных — это нормально, переменных ещё нет.
4. Проект → вкладка **Storage** → Create Database → **Blob** → access **Public**,
   имя любое (например `esque-media`). Vercel сам добавит в проект переменную
   `BLOB_READ_WRITE_TOKEN` — трогать её не нужно, код найдёт её сам.
5. Скопировать значение `BLOB_READ_WRITE_TOKEN` (Storage → хранилище → `.env.local` /
   Show secret) и положить локально в `.env` как `DEMO_BLOB_TOKEN=` — он нужен скрипту
   перелива, который запускается с вашей машины.

## Шаг 3. Перелить данные

В `.env` должны быть заполнены `DEMO_DATABASE_URL`, `DEMO_BLOB_TOKEN` и
`DEMO_ADMIN_PASSWORD` (пароль для входа в демо-админку — придумайте свой).

```
npm run export-demo -- --limit 150 --dry    # показать выборку, ничего не писать
npm run export-demo -- --limit 150          # залить
```
Нужен работающий локальный Docker (`docker compose up -d db minio`) — данные берутся оттуда.
Скрипт идемпотентен: повторный запуск обновляет, залитые файлы не перезаливает.

Выборка — по кругу разделов, свежие сверху (по 21-22 материала на каждый из 7 разделов),
чтобы главная, ленты и поиск выглядели заполненными. Простое «последние 150» перекосило бы
демо в «События».

Редиректы со старых URL намеренно не переносятся: они вели бы на материалы, которых в
выборке нет (404). Без них proxy уводит любой старый `.html` на главную.

## Шаг 4. Vercel: переменные и боевой деплой

Project → Settings → Environment Variables (все — Production):

| Переменная | Значение |
|---|---|
| `DATABASE_URL` | строка Neon |
| `NEXT_PUBLIC_SITE_URL` | URL, который выдал Vercel |
| `NEXT_PUBLIC_IS_DEMO` | `1` |
| `AUTH_SECRET` | новая длинная случайная строка |

`BLOB_READ_WRITE_TOKEN` уже добавлен на шаге 2 — по нему код сам переключается с MinIO
на Blob. Переменные `S3_*` на Vercel задавать НЕ нужно.

`NEXT_PUBLIC_SITE_URL` узнаётся только после первого деплоя, поэтому задаётся вторым
заходом с передеплоем — от него зависят canonical, OG и sitemap.

Затем Deployments → Redeploy.

## Известные ограничения демо

- **Загрузка фото в админке.** У Vercel лимит на тело запроса 4.5 МБ. Сценарий «дроп 12
  фото» упрётся в него: по 1-2 фото пройдёт, пачкой — нет. Лечится client upload прямо
  в Blob из браузера (в @vercel/blob это штатный режим) — отдельная работа, не делали.
- **Первый заход медленный.** Бесплатный Neon усыпляет базу при простое, пробуждение —
  несколько секунд. На проде это лечится платным тарифом.
- **Индексация выключена.** Перед превращением демо в прод снять `NEXT_PUBLIC_IS_DEMO`.

## Пароли

Репозиторий публичный, устройство входа видно всем, демо-пароль `esque-dev` лежит в сиде.
`export-demo` ставит всем пользователям пароль из `DEMO_ADMIN_PASSWORD` — задайте свой.
Почты те же: admin@esque.su / editor@esque.su / author@esque.su.
